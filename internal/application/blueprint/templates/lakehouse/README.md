# lakehouse

`cdc-to-lake` plus a Nessie catalog for the lake, an OpenLineage (Marquez)
lineage backend wired to both CDC connectors, and a second CDC pipeline
against an externally-hosted database reached through a managed
Connection.

```
Source(app-db) ──Binding(cdc)──▶ EventStream(app-events) ──Binding(sink)──▶ Dataset(raw-lake) → MinIO
                                                                                      │
                                                                            Catalog(lakehouse-catalog) → Nessie
Connection(legacy-db) → Source(legacy, external) ──Binding(cdc)──▶ EventStream(legacy-events)

Both CDC Bindings observe: Provider(lineage) → Marquez
```

Generated by `platformctl init lakehouse`. Every manifest here already
validates as written — no edits required to run `platformctl validate`
(the external Connection's `target` is a placeholder; validate never
dials out).

## Before you apply

1. **Build the S3 sink Connect image**:

   ```sh
   docker build -t datascape-s3sink-connect:local ./s3sink-image
   ```

2. **Fill in `.env`** with real credentials for the managed database, the
   object store, and the external legacy database.

3. **Point `connection.yaml` at your real legacy database** (edit
   `spec.target`) if you want the second CDC pipeline to actually run.
   Otherwise, comment out or delete `connection.yaml`, `source-legacy.yaml`,
   `eventstream-legacy.yaml`, `binding-cdc-legacy.yaml`, and the
   `ext-legacy-creds` entries in `secrets.yaml`/`.env` — the rest of the
   platform (the cdc-to-lake spine plus the catalog and lineage backend)
   stands on its own.

## Run it

```sh
platformctl validate .
platformctl apply    . --env-file .env --auto-approve
platformctl status   .
platformctl inventory .
```

`inventory --for trino` (once the D9/D10 compute-engine work lands) or
`--for spark`/`--for dbt` render paste-ready config against the catalog
and warehouse Dataset today.

Tear down:

```sh
platformctl destroy . --auto-approve
```

## What each addition is for

- **Catalog(lakehouse-catalog)**: the Iceberg REST catalog a compute
  engine (Spark, Trino) points at. `dataset-warehouse.yaml` is the bucket
  it catalogs; nothing writes to it yet in this blueprint — that's the
  compute engine's job, run outside platformctl.
- **Provider(lineage)**: a Marquez instance. Both CDC Bindings declare it
  under `metadata.observers`, so their connectors' native OpenLineage
  configuration points at it — lineage events show up in Marquez without
  any extra wiring.
- **Connection(legacy-db) + Source(legacy)**: the pattern for integrating
  a database platformctl doesn't manage. The Connection is the one place
  that knows where the database actually lives (`spec.target`); every
  consumer (here, the CDC connector) addresses the Connection's stable
  name instead.

See `examples/lakehouse/README.md` for the full worked version of this
pattern, including Dagster/Metabase pointers.
