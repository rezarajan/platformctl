# external-cdc

An externally-hosted database (one platformctl never creates or destroys)
reached through a managed Connection, captured by a managed Debezium
worker into a managed Redpanda topic. Use this when the source database
already exists — RDS, a shared staging instance, a coworker's box — and
you only want platformctl to own the CDC pipeline downstream of it.

```
external database (yours)
        │  (declared, not managed)
Connection(source-db) ── Source(source, external) ──Binding(cdc)──▶ EventStream(change-events)
```

Generated by `platformctl init external-cdc`. Every manifest here already
validates as written — no edits required to run `platformctl validate`,
because `connection.yaml`'s `target` is a placeholder
(`external-database-host:5432`) and validate never dials out.

## Before you apply

1. **Point `connection.yaml` at your real database**: edit `spec.target`
   (`external-database-host:5432`) to `<real-host>:<real-port>`, and
   `source.yaml`'s `postgres.database`/`schema` to match.
2. **Fill in `.env`** with the real database credentials.

## Run it

```sh
platformctl validate .
platformctl apply    . --env-file .env --auto-approve
platformctl status   .
platformctl inventory .   # the Connection's stable host-reachable address
```

The Connection's forwarder gives you one address to point tools at
(`platformctl inventory` shows it) regardless of where the database
actually lives — moving the database later means editing one line
(`connection.yaml`'s `target`), not every consumer.

Only tables listed on the CDC Binding's `options.tables` are captured
(`binding-cdc.yaml`); add more and re-apply to widen capture without
recreating any infrastructure.

Tear down:

```sh
platformctl destroy . --auto-approve
```
