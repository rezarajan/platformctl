# ADR 024 — Interactive composition: add / wire / expose as manifest compilers

**Status:** accepted (2026-07-22); scheduled as doc 08 E9 (core) and E10
(visual composer, optional/spin-off candidate). **Prompted by:**
project-owner direction — a "lego building blocks" experience: bootstrap
with `init`, then incrementally add a source+CDC+sink, add another source
reusing the existing sink at a different location, expose a source —
with flags or interactive prompts that draw candidates from the existing
graph.

## The one architectural rule

**Composition compiles to manifests; it never bypasses them.** Every
composition command parses the current directory's manifest set through
the same front-end as `validate`, computes a *manifest patch* (new YAML
files and/or additive edits, blueprint-quality: commented, `.env` keys
appended), writes it, and stops. The files remain the single source of
truth; `validate → lint → plan → apply` is unchanged. This keeps the
lego experience deterministic, diffable, reviewable, GitOps-compatible,
and automatically covered by every guardrail (ADR 011 validation, ADR
020 lints, ADR 021 policy). A composition layer that applied hidden
state would fork the source of truth — rejected outright.

## The verb quartet (terminology refined per the owner's invitation)

| Verb | Meaning | Emits |
|---|---|---|
| `init <blueprint>` | bootstrap a project (exists, E1) | a whole starter set |
| `add <composite>` | a new building block | the block's resources, reusing existing infrastructure where compatible |
| `wire <mode> --from <ref> --to <ref>` | connect two **existing** blocks | a Binding (+ any missing glue: worker Provider, EventStream) |
| `expose <Kind>/<name>` | a stable entrypoint to an existing block | a Connection (+ realizing Provider), scheme-selected: tcp → proxy, http(s) → ingress (+C8 TLS), mediated → mesh (post-H6) |

Composites are the incremental siblings of blueprints, drawn from the
same template stock: `source` (engine-prompted), `pipeline`
(source+CDC+broker+sink — the owner's "source-with-cdc-and-sink"),
`sink`, `catalog`, `monitoring`. The owner's scenario maps exactly:
`init` → `add pipeline` → `add pipeline` (prompts list the existing
broker and Dataset as reuse candidates; the Dataset offers a
prefix-override, emitting a second sink Binding to the same bucket at a
different location) → `expose Source/first-db`.

## Graph-aware reuse (the heart of the experience)

Before prompting, the command loads and resolves the existing set (the
`loadAndValidate` front-end, tolerant mode: an invalid set degrades to
best-effort candidates with a warning). Each composite declares its
*attachment points* (e.g. `pipeline` needs: a broker Provider — reuse or
create; a sink Dataset — reuse (with prefix override) or create; a
SecretReference — reuse or create+`.env` keys). Interactive mode renders
each attachment point as a select: existing candidates first (from the
graph, kind- and capability-filtered), then "create new…". Flag mode is
the same machine: `--broker existing:local-redpanda`,
`--sink existing:attendance-raw --sink-prefix other/`, `--source new
--engine postgres`. Every prompt has a flag; **non-TTY + incomplete
flags = hard error listing the missing flags** (CI-safe, never hangs).

## Interaction layer (research-informed, 2026-07)

- **Huh v2** (`charm.land/huh/v2`, the Charm forms library on Bubble Tea
  v2 — the ecosystem moved to `charm.land` vanity paths in v2, 2026-03)
  is the prompt layer: Select/Input/Confirm forms with a first-class
  accessible mode. The 2026 guidance is literal: use Huh for forms,
  Bubble Tea for custom interactions — we need forms.
- Dependency-class check (ADR 003 shape): pure-Go, ubiquitous, CLI-only.
  Confinement is architectural: charm imports are allowed ONLY under
  `cmd/platformctl` + `internal/cliutil` — enforced by an archtest, so
  the composition *engine* (candidate computation, patch generation)
  stays TUI-free and testable headless in `internal/application/compose`.
- A full visual composer (live graph canvas, drag-to-wire — true Bubble
  Tea territory) is E10: explicitly optional and a natural spin-off
  project boundary; the engine/TUI split above is exactly the seam it
  would build on. Not designed here.

## Quality bar

Generated output must meet the blueprint bar: the resulting set
validates green with zero edits, **lints clean** (H1 — composition and
lints co-evolve: a composite that emits a DL-flagged topology is a bug),
idempotent generation (`add` twice without answers changing = second run
proposes no changes), `--dry-run` prints the exact files/diffs without
writing, and generated files carry a provenance comment
(`# generated by platformctl add pipeline; edit freely — files are the
source of truth`). Name collisions prompt for a new name; nothing is
ever overwritten silently.

## Out of scope, with reasons

- Applying anything: composition writes files; the user applies. No
  `--and-apply` sugar initially — it would blur the one rule for minor
  convenience.
- Editing arbitrary existing resources (`platformctl edit`): text
  editors and the reference docs own that; composition only *adds* and
  *wires*.
- A long-running daemon/watch composer: contradicts the one-shot posture
  (doc 09 §4.1).
